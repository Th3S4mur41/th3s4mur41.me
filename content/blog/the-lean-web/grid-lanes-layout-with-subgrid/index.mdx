# Grid Lanes, Subgrid, and No Extra Wrappers

![Comic-style illustration of Michaël Vanderheyden slashing the JavaScript logo with a katana](../michael-slashing-the-JS-logo-with-katana.jpg)

The goal of this series is simple: build on modern HTML and CSS to remove old workarounds and JavaScript fixes, and build modern, robust UIs through progressive enhancement.

For the first post, I wanted to focus on something that has been bugging me on my own page since I added it and wanted to test a better solution.
For article pages, I want body content centered for readability, with occasional full-bleed elements like hero images, quotes, or media blocks. The classic approach is a grid-based full-bleed layout, like Josh Comeau's write-up on [Full-Bleed Layout Using CSS Grid](https://www.joshwcomeau.com/css/full-bleed/). A page-level grid plus subgrid is already a solid base for this and helps align content across sections without structural div soup.

The challenge shows up on wider screens. That's where I want to move secondary content, like a table of contents and reactions, into a sidebar. Historically, this is where wrappers and magic numbers creep in.
I wanted a version that still works with markdown-generated content, with no extra container choreography in the markup.

## Start from content, not containers

The article body is generated from markdown, so the structure is naturally linear: headings, paragraphs, media, and a footer.

I use a rehype plugin to generate and inject a table of contents after the h1.
Reactions are added by Astro logic at build time in the footer. No wrapper around "main content" and "sidebar" blocks.

```html 
 <article>
    <h1>...</h1>
    <nav class="table-of-contents">...</nav>
    <p>...</p>
    <p>...</p>
    ...
    <footer>
      <div class="reactions">...</div>
    </footer>
  </article>
```

That is ideal for Lean Web: semantic HTML first, then layout as enhancement.
This is also the structure this very page you are looking at is built on.

## Where the old workaround starts

On larger screens, side space is often underused, so moving the table of contents to the sidebar was a natural next step.

Moving across columns is easy. Rows are where things get awkward.
In a grid, the table of contents shares the first row with the h1 or the first paragraph, this row grows to match the table of contents height which is usually bigger than the content.
As a result, the first element is then followed by a weird spacing.

The table of contents height depends on content, so hard-coding row spans is guesswork.
The classic workaround is "magic number" row spans: big enough for most cases, hopefully not too big to potentially break things.

While this solution works to some extent, allowing me to move the table of contents to the sidebar without changing the HTML structure, it is still a workaround and I don't like magic numbers.

## The solution: grid lanes

<baseline-status featureId="grid-lanes"></baseline-status>

Grid lanes let you define named lanes and place items so they do not force sizing behavior across lanes the way traditional track sharing does.

ADD SCREENSHOT HERE

In practice, I define a content lane and a sidebar lane, then place the table of contents in the sidebar lane.
Now the first content element keeps its natural height, and the old row-span workaround disappears.
That works great if the columns are defined in the the article, but what if like on this page, the columns are defined in a parent?

### What about subgrid?

At first glance, it feels like moving to lanes might block subgrid.
But the non-lane axis still behaves like regular grid, so subgrid remains useful for content alignment.

> [!WARNING]
> At the time of writing, Chrome supports grid lanes behind experimental flags, but the combination with subgrid is not fully there yet.
> This pattern should be treated as progressive enhancement knowing that it might break with experimental implementation

## More content in the sidebar

Now the the table of content is using the exact height it needs, we could think about moving more things to the sidebar.
On this page, the natural next item would be the article footer or the reactions.

With `grid-lanes`, moving the footer to the sidebar is straightforward and it gets place right below the table of contents where I would expect it.

## Make sidebar elements stick without guessing offsets

I usually keep the table of contents sticky so readers can navigate while scrolling.
That part is straightforward and doesn't change with `grid-lanes`

The trickier part now is the footer below it. I do not know the table of contents' height ahead of time, so a fixed top offset is just another magic number waiting to fail.

### The dream solution: sticking to an anchor

The perfect solution to this problem would be to set `position: sticky` to the footer but instead of a fixed number for the `inset-block-start` (or `top`), we would use the table of contents as an anchor.
The solution would be:

```css
nav.toc {
  anchor-name: --toc;
}
footer {
  grid-column: sidebar;
  position: sticky;
  inset-block-start: anchor(--toc end);
}
```

Unfortunately, combioning `position: sticky` with anchor positioning, isn't currently possible.

### The second shift: anchor positioning

<baseline-status featureId="anchor-positioning"></baseline-status>

So instead, I can use anchor positioning to place the footer right below the table of contents, and it will stay there as the content grows.

This solves the happy path cleanly, but there is still a caveat.
If sidebar content becomes taller than the viewport, the anchored element can be pushed down and overflow the page's footer or worse bleed out of the viewport.
In this case unfortunately, we'll have to go back to our magic number workaround and add the estimaged height of the footer as a `margin-block-end` of the table-of contents.

## Why this belongs in the Lean Web series

This pattern is not a silver bullet, and wrappers are still sometimes the pragmatic choice.

But it captures a Lean Web principle I care about: when the platform grows, we should remove complexity, not just move it around.

Here, that means:

- keeping markdown-native HTML structure
- avoiding wrapper-only layout elements
- removing row-span and offset magic numbers
- shipping it as progressive enhancement while browser support matures

If you are experimenting with similar patterns, I would love to hear what worked, what broke, and what fallback strategy you picked.
