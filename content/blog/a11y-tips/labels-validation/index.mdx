# Labels: Automatic Checks and Manual Validation

![Comic-style illustration of a bearded developer wearing a baseball cap and samurai-themed shirt, working on a laptop with a samurai sticker. To the side, accessibility icons and the hashtag #A11yTips promote practical accessibility tips for developers.](../Practical_Accessibility_Tips_for_Developers-v2.jpg)

Labels are essential for screen reader users and overall accessibility. While automated tools can verify the presence of labels on interactive elements, assessing whether a label is meaningful and contextually appropriate usually requires manual testing.

[Testing with screen readers](https://medium.com/r/?url=https%3A%2F%2Fwebaim.org%2Farticles%2Fscreenreader_testing%2F) can be challenging to use for developers and testers who are not regular users. Alternative methods include:
- Using DevTools accessibility tree to inspect labels and roles.
- Employing voice navigation features to test if elements are reachable and operable by voice commands.

## DevTools Accessibility Tree

For developers who aren't regular screen reader users, browser DevTools offer a practical way to inspect how labels and roles are exposed to assistive technologies. Both Chromium-based browsers (like Edge and Chrome) and Firefox include an **Accessibility Tree** view that mirrors what screen readers perceive.

This tree shows the semantic structure of your page - including roles, names, states, and relationships - allowing you to verify whether interactive elements are properly labeled and accessible.

### How to access the Accessibility Tree

#### Edge / Chrome:
1. Open DevTools (`F12` or `Ctrl+Shift+I`)
2. Open DevTools settings though the 3-dot menu
3. Go to the experiments tab and activate the **Enable full accessibility tree view in Element panel** option
4. Go to the **Elements** tab
5. Click the floating Accessibility button in the top right of the pane

#### Firefox:
1. Open DevTools (`F12` or `Ctrl+Shift+I`)
2. Go to the **Accessibility** tab directly
Navigate the tree or select elements to inspect roles, names, and states

The screenshot above compares both views using a demo from [this CodePen](https://codepen.io/th3s4mur41/pen/MYKGORX), showing how a checkbox labeled "Read" is exposed in the accessibility tree. You can see its role, name, and state - all critical for screen reader compatibility.

## Voice Navigation

Screen readers aren't the only way to validate whether labels are meaningful and usable. Voice navigation tools let you interact with a page the way users relying on speech commands would - asking the system to "click," "open," or "select" elements by name. This is a powerful way to check if your labels are both present and practical in real-world use.

### Why it matters:
- If a button or link can't be activated by its label, it's a sign the label isn't descriptive enough.
- Voice navigation highlights issues that automated checks often miss, such as ambiguous or duplicated labels.
- It helps ensure that your interface works for users who rely on voice control for accessibility or convenience.

### How to test with voice navigation:

#### Windows (Voice Access):

1. Enable Voice Access in Accessibility settings.
Use commands like "Click Read" or "Open Settings" to test if elements respond correctly.

#### macOS (Voice Control):

1. Turn on Voice Control in System Preferences → Accessibility.
2.Try commands such as "Click Log In" or "Select checkbox".

#### Mobile (iOS / Android):
iOS: Use Voice Control or VoiceOver with speech

> [!NOTE]
> Voice navigation isn't just for end users with accessibility needs. Developers themselves may benefit from it at some point in their careers. For example, those experiencing hand mobility issues such as carpal tunnel syndrome can use voice commands as an alternative way to interact with the UI. Building with accessibility in mind ensures that these options are available when you need them most.
